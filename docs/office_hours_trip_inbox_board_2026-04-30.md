## Product Direction

Trip planning leaks across too many surfaces. The user finds a flight and leaves it in a phone browser tab, finds a hotel in Booking, asks ChatGPT for ideas in a separate conversation, saves restaurants or notes somewhere else, then loses half the work or has to reconstruct it later.

The product should make ChatGPT the personal travel agent, concierge, and adviser, while MCP Apps UI turns the conversation into a persistent visual trip workspace.

The first job is not "generate an itinerary." The first job is "do not lose the planning work I already did."

## Recommended Approach

Build **Trip Inbox + Trip Board**.

The Trip Inbox captures messy travel fragments: links, notes, booking options, restaurant ideas, documents, and questions.

The Trip Board turns those fragments into structured planning state: open decisions, shortlisted options, booked items, itinerary draft, and missing pieces.

This is the smallest version that captures the real product:

1. Nothing gets lost.
2. The mess becomes a visible plan.

## V1 Scope

### Trip Inbox

Purpose: safe landing zone for unprocessed planning work.

Item types:

- Flight option
- Train or transport option
- Hotel or stay option
- Restaurant
- Activity
- Document or reservation
- Chat note
- Question
- Constraint

Each item should preserve:

- Raw text or URL
- Source label
- Detected type
- Trip association
- Status: `inbox`, `shortlisted`, `booked`, `rejected`, `needs_review`
- Optional date or day
- Notes from ChatGPT

### Trip Board

Purpose: turn raw fragments into decisions and plan structure.

Initial lanes:

- Open Decisions
- Shortlisted
- Booked / Confirmed
- Itinerary Draft
- Missing Pieces

The board should make it obvious what the user has already found, what is decided, and what still needs work.

### ChatGPT Interaction

Target commands:

- "Save this hotel to my Barcelona trip."
- "Organize my inbox into the board."
- "Compare the two hotels I saved."
- "What am I still missing?"
- "Turn these shortlisted activities into a 3-day itinerary."
- "Use weather to decide which activities should be indoors."
- "Generate my packing list from the itinerary."

## Original Success Criteria

The first version is done when the builder can use it for a real upcoming trip and:

- Save at least 10 messy travel fragments into one trip.
- See those fragments classified in a Trip Inbox.
- Promote or organize fragments into a Trip Board.
- Ask ChatGPT what is missing and get a structured answer based on saved trip state.
- Compare at least two hotel or transport options from saved items.
- Generate a draft itinerary from saved board items.
- Use existing weather or packing MCP functionality from the trip context.

## Implementation Delta

The code has now implemented the core of the office-hours plan with two important changes:

- The design doc proposed SQLite first. The code now uses Postgres through `DATABASE_URL`, with `TRIP_STORE_BACKEND=file` available for temporary smoke testing.
- The design doc proposed a future `/mcp/trips` endpoint. The code now uses one unified endpoint, `/mcp/travel-agent/`, so ChatGPT can connect to one MCP app for trip state plus weather, travel tips, activities, and packing.

Implemented pieces:

- `services/trips.py` contains trip and item persistence, dedupe, classification, board grouping, and summary logic.
- `mcp_servers/travel_agent_server.py` exposes `create_trip`, `add_trip_item`, `list_trip_inbox`, `update_trip_item_status`, `get_trip_board`, and `get_trip_summary`.
- `mcp_servers/widgets/trip_inbox_v1.html` renders the Trip Inbox widget.
- `mcp_servers/widgets/trip_board_v1.html` renders the Trip Board widget.
- `/mcp/travel-agent/` is mounted in the FastAPI app.
- Tests cover trip storage, unified MCP tool output, widget resources, and mounted endpoints.
